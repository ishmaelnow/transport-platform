# Resend Account Recovery and Credential Rotation Plan

## Purpose

This runbook records the current Resend configuration risk and provides a zero-downtime recovery plan.
The application is still an MVP, so the immediate decision is to preserve the working integration while
development continues. Complete the recovery and credential rotation before public launch or meaningful
customer onboarding.

This plan does not require an architectural or business-logic change.

## Current State

Verified on July 23, 2026:

- Production invitation emails send successfully through the Admin application.
- Invitation links and the Admin webhook URL are reachable.
- The working `RESEND_API_KEY` and `RESEND_WEBHOOK_SECRET` are stored in the Admin Vercel project.
- The Resend account currently accessible to the project owner does not show the historical API key,
  email activity, or original team.
- Only one team is visible in that accessible Resend account.
- The original development laptop is damaged or unavailable.
- `eshapp.com` uses Netlify's authoritative DNS nameservers.
- `admin.eshapp.com` is published through Netlify DNS and points to Vercel.
- The following public DNS records for the original Resend integration are active:
  - SPF at `send.eshapp.com`
  - MX feedback routing at `send.eshapp.com`
  - DKIM at `resend._domainkey.eshapp.com`
- No public DNS records currently exist for the proposed replacement domain `mail.eshapp.com`.

The registrar, authoritative DNS provider, application host, and email provider are separate concerns:

```text
Domain/DNS: Netlify
Application hosting and runtime variables: Vercel
Transactional email: Resend
```

## Current Risk

The application depends on a valid Resend key whose owning Resend team is not currently accessible. The
key may continue working because Resend keys do not expire automatically, but it could be revoked without
warning by whoever controls the original team.

Until rotation is complete:

- Do not delete or replace the working Vercel values.
- Do not delete the active SPF, MX, or DKIM records.
- Monitor invitation-send failures and webhook errors.
- Keep real secrets out of Git, documentation, screenshots, tickets, and chat.

## Recovery Evidence

Collect the following without exposing secrets:

- Email address that received the Resend welcome message.
- Welcome-message date and time.
- Exact date, time, timezone, sender, recipient, and subject of a recent successful invitation.
- The recent message's full headers, including `Message-ID`, from **Gmail → More → Show original**.
- Names of all possible Google, GitHub, and email/password identities used with Resend.
- Names of all visible Resend teams.
- Admin Vercel project name and the names—not values—of its Resend environment variables.
- Only a partial API-key identifier if Vercel or Resend displays one; never disclose the complete key.

## Phase 1: Attempt Account Recovery

1. Sign out of Resend.
2. Use a private browser window.
3. Try **Log in with Google** using the account that received the welcome message.
4. Try any other likely Google, GitHub, or email/password identity.
5. Inspect every available team and its **Emails**, **Logs**, **API Keys**, **Domains**, and **Webhooks**.
6. Clear dashboard filters and verify that recent successful invitations are absent.
7. Contact Resend support with the recovery evidence above.
8. Ask Resend to identify or restore access to the team that sent the recent invitation.

Do not send the complete API key to support through ordinary email. Use only Resend's authenticated support
process if proof of key control is required.

### If access is recovered

1. Confirm the domain, API key, webhook, and recent email activity.
2. Add a second trusted administrator or recovery method where appropriate.
3. Create a replacement API key and webhook secret.
4. Continue with the Vercel rotation and validation steps below.
5. Revoke the old credentials only after the replacement passes end-to-end testing.

## Phase 2: Replacement Setup if Recovery Fails

Use the Resend account that is under confirmed control.

### 1. Add a dedicated sending subdomain

Add:

```text
mail.eshapp.com
```

The intended sender is:

```text
ESH Platform <onboarding@mail.eshapp.com>
```

If Resend reports that the domain is already registered, stop and ask Resend support to release or transfer
it. Do not remove the working DNS records as a workaround.

### 2. Add the generated DNS records

Add every record generated by Resend under **Netlify → DNS → eshapp.com**. Copy the exact record type,
hostname, value, region, and MX priority shown by Resend.

Do not:

- Add the records only to a non-authoritative Vercel DNS zone.
- Delete or modify `admin.eshapp.com`.
- Delete the original `send.eshapp.com` or `resend._domainkey.eshapp.com` records.
- Reuse example DKIM, SPF, or MX values.

Wait until Resend reports `mail.eshapp.com` as verified.

### 3. Create new credentials

1. Create a domain-restricted sending key named `esh-platform-production`.
2. Copy the API key once and store it in an approved password or secret manager.
3. Create a webhook for:

```text
https://admin.eshapp.com/api/webhooks/resend
```

4. Subscribe to:

```text
email.delivered
email.delivery_delayed
email.bounced
email.failed
email.suppressed
```

5. Copy and securely store the webhook signing secret.

## Phase 3: Zero-Downtime Vercel Rotation

Update only the Admin Vercel project:

```env
RESEND_API_KEY=re_new_value
RESEND_WEBHOOK_SECRET=whsec_new_value
INVITATION_FROM_EMAIL=ESH Platform <onboarding@mail.eshapp.com>
```

Apply the values to Production. Apply them to Preview only if preview deployments are allowed to send
email. Do not add Resend secrets to the Rider or Driver projects.

After saving the values:

1. Redeploy the Admin project.
2. Confirm the deployment is `Ready`.
3. Send a new invitation to a controlled inbox.
4. Confirm the Admin UI initially records `pending`.
5. Confirm the email appears in the new Resend dashboard.
6. Confirm the recipient receives the invitation.
7. Confirm the invitation link starts with `https://admin.eshapp.com/invite/accept`.
8. Confirm Resend successfully delivers webhook events.
9. Confirm the Admin UI updates the invitation to `sent` after delivery.
10. Check Vercel function logs for unexpected errors.

## Rollback

Keep the previous Vercel values in a secure recovery record until the new integration passes all validation
steps.

If the new configuration fails:

1. Restore all three previous Resend values together.
2. Redeploy Admin.
3. Send a controlled test invitation.
4. Confirm the original integration works.
5. Diagnose the new domain, key, sender, or webhook before attempting rotation again.

Never combine values from different configurations. In particular, a new webhook secret cannot verify
events from the old webhook.

## Completion Criteria

Recovery is complete when:

- The project owner controls the Resend team used by production.
- Production uses a newly created and securely stored API key.
- The webhook signing secret belongs to a visible, controlled webhook.
- The sending domain is verified in the controlled account.
- A production invitation is delivered and visible in the controlled Resend dashboard.
- Signed webhook delivery updates the corresponding invitation status.
- The inaccessible credentials are revoked or confirmed unusable.
- Operational ownership and recovery information are stored securely.

## Deferred DNS Consolidation

Moving authoritative DNS from Netlify to Vercel is optional and separate from Resend recovery. Do not
combine the two changes.

If consolidation is later approved:

1. Inventory every live Netlify DNS record.
2. Recreate the complete zone in Vercel.
3. Verify application, Supabase, email, DKIM, SPF, MX, and any domain-verification records.
4. Change authoritative nameservers only after the Vercel zone is complete.
5. Monitor DNS propagation and application/email health.
6. Retain a tested rollback plan until propagation is complete.
